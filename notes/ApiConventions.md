# API Conventions

This note captures the current conventions used in the Support Ticket API so behavior stays consistent as the project grows.

## Route Style

The API uses resource-oriented routes under `/api`.

### Current route patterns
- `/api/health`
- `/api/tickets`
- `/api/tickets/{id}`
- `/api/tickets/{id}/status`
- `/api/tickets/{id}/comments`
- `/api/tickets/{id}/history`

### Route conventions
- top-level resources use plural nouns (for example, `tickets`)
- nested resources are grouped under their parent ticket when they belong to a specific ticket
- route parameters use `id` for ticket identifiers
- route constraints are used where appropriate (for example, `{id:int}`)

---

## Request / Response Conventions

### Request models
- request DTOs are used for client input instead of binding EF Core entities directly
- server-owned values are not accepted from the client when they should be controlled by the API
  - examples:
    - ticket ID
    - comment ticket ID
    - timestamps
    - status history timestamps

### Response models
- response DTOs are returned instead of exposing EF entities directly
- response DTO mapping is centralized with private mapping methods inside the controller where possible

---

## Status Conventions

### Allowed statuses
- `Open`
- `InProgress`
- `Resolved`
- `Closed`

### Status rules
- incoming status values are trimmed
- incoming status values are normalized to canonical stored values
- invalid status values are rejected with `400 Bad Request`
- status changes create a `StatusHistory` entry when the value actually changes

---

## Priority Conventions

### Allowed priorities
- `Low`
- `Medium`
- `High`
- `Critical`

### Priority rules
- incoming priority values are trimmed
- incoming priority values are normalized to canonical stored values
- invalid priority values are rejected with `400 Bad Request`

---

## Input Handling Conventions

### Title
- required
- trimmed before storage
- whitespace-only values are rejected
- maximum length is enforced by DTO validation

### Description
- required
- trimmed before storage
- whitespace-only values are rejected
- maximum length is enforced by DTO validation

### Comment body
- required
- trimmed before storage
- whitespace-only values are rejected
- maximum length is enforced consistently

---

## Error Handling Conventions

### `404 Not Found`
Used when a route-targeted resource does not exist.

Examples:
- ticket ID does not exist
- comments requested for a missing ticket
- history requested for a missing ticket

### `400 Bad Request`
Used when the client provides invalid input.

Examples:
- invalid status
- invalid priority
- invalid author user ID
- invalid created/assigned user ID
- empty or whitespace-only content

### Current error style
- error messages should be short, direct, and specific to the invalid input
- similar validation failures should use similar wording across endpoints

---

## Data / Time Conventions

- timestamps are generated by the server
- timestamps use UTC
- persisted state should reflect actual application state
- save/update workflows should avoid storing unnormalized or misleading values

---

## Query Behavior Conventions

### Ticket listing
`GET /api/tickets` currently supports filtering by:
- `status`
- `priority`
- `createdByUserId`
- `assignedToUserId`

### Query conventions
- optional filters should narrow the result set when provided
- when filters are omitted, the endpoint returns all matching resources
- list results should be returned in a consistent order

---

## Controller Conventions

- controllers focus on request handling, validation, persistence calls, and response generation
- repeated DTO mapping logic should be extracted into private helper methods when useful
- normalization logic should be applied consistently before persistence
- small helper methods are preferred over repeated inline transformation code

---

## Development / Testing Conventions

- features should be verified in Swagger after implementation
- important DB changes should be spot-checked in SQL Server Object Explorer
- bug fixes and feature behavior should be easy to verify with real stored data
- route, input, and persistence behavior should be tested together, not in isolation

---

## Current Philosophy

This API is being built as a practical support-ticket style backend project.

The goal is not to maximize feature count as quickly as possible. The goal is to build a project that demonstrates:
- clear API design
- consistent input handling
- maintainable controller code
- practical database usage
- support and maintenance mindset
